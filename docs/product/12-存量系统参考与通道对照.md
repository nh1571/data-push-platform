# 存量系统（pythonProject4）→ 新中台：可参考点与通道对照

> 来源：`20_项目/现有钉钉推送系统-pythonProject4分析.md`  
> 态度：**仅供参考**。旧系统满足业务但难用；新中台继承「能力与管线」，不继承「硬编码脚本堆叠」。

---

## 1. 旧系统在干什么（一句话）

爱眼数据中心侧：**多院 HIS/SQL Server + Doris + Oracle 取数 → HTML 表格成图 → Markdown/图片/文件/互动卡片 → 钉钉单发/群发/Webhook**；crontab 定时；约 150+ 任务脚本。

**证明了需求真实且规模不小**——新中台要对齐的是这类业务，不是玩具 demo。

---

## 2. 旧系统钉钉「推送通道」二维模型（核心参考）

旧代码用 **通道类型 × 消息形态** 二维分发（工厂 + `single_mode`），**不是**「只有一个钉钉」。

### 2.1 通道类型（`dingtalkFactory` / 策略子类）

| 旧 type | 实现 | 含义 | 新中台建议 mode |
|---------|------|------|-----------------|
| **single** / `singleshot` | 官方 SDK `DingtalkSample(type=0)` → **BatchSendOTO** | 机器人 **单发**（按 user_ids） | `dingtalk.openapi_oto_robot` |
| **orggroupsend** | 官方 SDK `DingtalkSample(type=1)` → **OrgGroupSend** | 机器人 **群发**（open_conversation_id） | `dingtalk.openapi_group_robot` |
| **masstexting** | 社区库 `DingtalkChatbot` | **Webhook 机器人** markdown | `dingtalk.webhook_robot` |
| **masstexting_basics** | 内部 HTTP 中转 `pushMakDownToQun` | 经业务中台转发的群 Markdown | 可选 `dingtalk.internal_relay` 或一期不做 |

### 2.2 消息形态（`single_mode` 二次分发）

| single_mode | 含义 | 新中台对应 |
|-------------|------|------------|
| **simple** | 纯文本/Markdown | text / markdown 部件 |
| **image** | 图片消息 | image 部件 + media 上传 |
| **file** | 文件 | file 部件 |
| **un** | 互动卡片（含取消订阅等） | interactive card（后置） |
| **care** | 关怀卡片 1–6 按钮 | 卡片模板（后置） |
| **atun** | @所有人 等 | 通道配置项 |
| **Vertical_triad** | 三联按钮 | 卡片（后置） |

### 2.3 底层三类发送引擎（与开放平台一致）

| 引擎 | 技术 | 用途 |
|------|------|------|
| Webhook | `dingtalkchatbot` | 群自定义机器人，快，**图能力弱** |
| 官方 OpenAPI | `alibabacloud_dingtalk` + Token 刷新 | **单发 OTO / 群发 OrgGroup**，生产主力 |
| 内部中转 | HTTP 调业务 PushMessageController | 历史/特殊链路，新系统可后置 |

**Token：** 提前 100s 刷新；每实例自管——新系统应收口为 **共享 Token 缓存**（Redis/进程内）。

### 2.4 其它钉钉周边能力（旧系统有、新系统可分期）

| 能力 | 旧文件 | 新系统优先级 |
|------|--------|--------------|
| 已读/互动查询 | get_isread.py | P2 |
| 撤回单发/群发 | recall_oto / recall_otg | P2 |
| 退订/点赞回调 | 硬编码 bonus URL | P2 产品化 |
| processQueryKey 落库 | dim_push_* 表 | P1 对齐 JobRun/Delivery |
| 用户 >20 拆批 | split_userid | **P0 实现时带上** |

## 3. 图片管线（旧系统最有价值的业务能力）

```text
SQL → DataFrame
  → PyEcharts Table → HTML
  → 注入 CSS（独立 css/ 目录）
  → 条件着色（百分比红绿）
  → wkhtmltoimage → PNG
  → 钉钉 image 模式发出
```

**可继承：**

- **HTML 中间层 + CSS 主题**（比纯 Pillow 表格更贴业务「好看表」）  
- 样式与代码分离（`push/css/`）  
- 条件着色规则可产品化进模板  

**勿继承：**

- `os.system(wkhtmltoimage)` 无返回值检查  
- 每个任务手写一套 chats/getdata/md 三件套  

**新中台路径建议：**

1. 短期：Pillow 模板（已有 report/alert/kpi 方向）  
2. 中期：对齐旧管线 **HTML+CSS+截图**（Playwright/wkhtmltoimage 二选一），模板可配置  
3. 发图走 **openapi 单发/群发**，不走 Webhook

## 5. 旧系统「满足需求但不好用」的根因 → 新系统要避开

| 痛点 | 表现 | 新中台对策 |
|------|------|------------|
| 任务即代码 | 150+ 独立 .py + crontab | 配置化 PushJob + 编辑器 |
| 改 main 分支地狱 | if-elif mode | 统一 Run API + 调度 |
| 凭证/人/群硬编码 | 散落各文件 | Channel / 受众配置化 |
| 重复实现 | care/file 单群两套 | Message 一次构建，Channel 只适配 |
| 无测试 | 全靠线上 | pytest 已铺，继续保持 |
| 新旧模式并存 | main.py vs work/main | 单一工作台路径 |
| 开发体验 | 样板代码复制 | UI 编辑 + 试推 |

**旧系统证明「能力集合」对；「交付形态」错。新系统 = 能力对齐 + 形态产品化。**

## 7. 前行优先级（吸收参考后）

1. **通道模型与旧二维对齐**  
   - Mode：webhook | openapi_group | openapi_oto | work_notice | …  
   - Message 形态：simple/image/file/card 独立于 Mode  
2. **成图对齐业务**  
   - 先打磨模板成图；再引入 HTML+CSS 截图管线（借鉴旧 css/ 思路）  
3. **数据源补 SQL Server**（院区刚需，旧系统主力）  
4. **JobRun 元数据对齐 processQueryKey 类追踪**（为撤回/已读留钩子）  
5. **绝不**把 150 个脚本迁成 150 个页面字段——用编辑器 + 任务配置覆盖
