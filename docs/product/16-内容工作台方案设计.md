# 内容工作台方案设计

> **目标一句话**：像用「场景模板 + 内容块」拼推送，而不是填一长串配置表单；  
> 操作路径要短，产出却能是「文案 + KPI + 好看表图 + 附件」的组合载体。  
> 关联：[[13-内容生产管线-核心关注]] · [[15-交互理念重设计]] · [[05-UI视觉设计]]  
> **2026-07-16 修正**：用户要求画布/组件/帆软式组合 → 核心架构见 **[[17-组件画布-内容生产核心设计]]**。本文「竖向块」保留为画板 **Flow 布局** 的一种模式，不再作为终态上限。

---

## 0. 问题定义

### 0.1 现状

当前工作台 = **单 SQL + 少量 Design 字段 + 三模板下拉 + MD/图二选一**。  
能跑通链路，**撑不住**院区日报、告警、多段指标、文图混排等真实推送。

### 0.2 产品矛盾（要同时成立）

| 极 | 含义 | 失败形态 |
|----|------|----------|
| **简单操控** | 新人 3 分钟能出一条可预览推送 | 画布/自由排版 → 培训成本爆炸 |
| **丰富美观** | 观感对齐旧 HTML 表图，结构可多样 | 只有一个 Markdown 框 → 业务不愿迁 |

**设计原则：复杂度放在「模板与渲染引擎」，不放在「用户每次操作」。**

## 1. 主流方案对标（我们学什么 / 不学什么）

| 参照物 | 核心模式 | 值得学 | 不直接照搬 |
|--------|----------|--------|------------|
| **Notion** | 一切皆块；`/` 插入；拖拽排序；所见即所得 | 块模型、插入菜单、块级操作 | 完整文档协作、无限块类型、协同锁 |
| **邮件构建器**（Mailchimp / 企业通知类） | 竖向模块堆叠；预置模块；主题色 | **竖向堆叠**最适合「一条推送消息」；模块预览 | 响应式邮件 HTML 全套 |
| **Canva / 海报** | 模板先行；少参数出成片 | **场景模板一键开工**；主题包 | 自由画布、像素级拖拽 |
| **Metabase / 看板** | 查询绑定可视化 | 数据→可视化的绑定心智 | 完整 BI；我们终点是**消息载体**不是仪表盘 |
| **飞书/钉钉消息卡片** | 卡片结构固定、字段填空 | 输出形态克制（IM 不是网页） | 绑死某一家卡片 JSON（我们要 Message 抽象） |
| **旧系统 pythonProject4** | DF → HTML/CSS → 截图 → MD 拼装 | **好看表**主路径；条件着色；样式资产独立 | 每任务一套脚本 |

### 1.1 合成结论（我们的定位）

```text
不是 Notion 全量编辑器
不是 Canva 自由画布
不是纯 BI 看板

而是：
  「邮件式竖向模块」+「Notion 式块操作」+「Canva 式场景模板」
  +「数据绑定」+「成图/富文本双出口」
```

**隐喻（用户心智）**：  
> 像拼一份**竖版简报**，每一段是一种模块；数据从 SQL 来，样式多半由模板负责。

## 2. 体验目标

### 2.1 成功标准（可验收）

| # | 场景 | 期望 |
|---|------|------|
| A | 空白开工 | 选「院区日报」模板 → 换数据源/SQL → 预览已是完整结构（无需从零搭） |
| B | 微调 | 改标题文案、关某一块、换主题色 → 实时预览 |
| C | 丰富 | 同一任务内可同时有：文案块 + KPI 块 + 表图块（可选附件块） |
| D | 简单 | 80% 任务 **不写 CSS、不拖绝对坐标**；高级项折叠在「块设置」 |
| E | 美观 | 表图默认质量 ≥ 旧系统「能看的表」；主题统一不花 |

### 2.2 反目标（明确不做）

- 像素级自由画布 / 无限嵌套容器  
- 完整 Notion 协同与权限体系  
- 在工作台内写 Python  
- 为每个钉钉 Mode 单独做一套排版 UI

## 3. 信息架构：三层模型

```text
┌─────────────────────────────────────────────────────────────┐
│  L0 场景模板 SceneTemplate（「我要发什么类型」）               │
│     日报 / 告警 / KPI 简报 / 纯文案 / 空白                     │
│     → 一键装载：数据槽 + 默认块序列 + 主题                      │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  L1 推送草稿 PushDraft（工作台编辑对象）                       │
│     name · 数据槽(s) · blocks[] · theme · channels · 调度…   │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  L2 消息载体 Message（运行时产物）                             │
│     parts[]: text_md | image | file | card …                │
│     通道只消费 Message，不理解块编辑器                         │
└─────────────────────────────────────────────────────────────┘
```

| 层 | 谁改 | 频率 | 存储 |
|----|------|------|------|
| 场景模板 | 产品/管理员预置；后期可自定义 | 少 | 代码或 `scene_templates` 表 |
| 推送草稿 | 业务日常 | 高 | `push_jobs.render_spec`（演进为 draft schema） |
| Message | 引擎生成 | 每次运行 | JobRun 快照 / 预览临时 |

**与现架构兼容**：现有 `design` 可视为「单块/固定结构」的特例；演进为 `blocks[]` 后，`design_to_parts` 升级为 `draft_to_message`。

## 4. 交互模型：简单操控怎么做

### 4.1 主路径（黄金路径，必须极短）

```text
打开内容工作台
  → 【选场景】或【空白 + 从模板库插入】
  → 【绑数据】选数据源 + SQL（可先运行取数）
  → 【看预览】右侧即最终观感（图/MD）
  → 不满意则点左侧某一块微调（文案/是否显示/样式预设）
  → 【试推】/【保存为任务】
```

**默认不强迫**用户先理解「块协议」：场景模板已经替他排好块。

### 4.2 三种难度档（渐进披露）

| 档 | 用户动作 | 覆盖 |
|----|----------|------|
| **L0 填空** | 换 SQL、改标题、选主题 | ~60% 标准任务 |
| **L1 调块** | 显示/隐藏块、改块内文案、换表样式预设 | ~30% |
| **L2 编排** | 增删块、排序、多数据槽、条件显示 | ~10% 复杂任务 |

UI 默认停在 L0–L1；L2 用「+ 添加模块」「⋮⋮ 拖拽」暴露，不塞进首屏表单。

### 4.3 关键手势（学 Notion / 邮件构建器，收敛版）

| 手势 | 行为 |
|------|------|
| **场景入口** | 空白页大卡片：日报 / 告警 / KPI / 空白 |
| **+ 添加模块** | 底部或块间：文本 / 指标卡 / 数据表图 / 分割说明 / 附件 |
| **块把手** | 悬停左侧：上移下移 / 复制 / 删除 / 折叠设置 |
| **点击块** | 右侧或块内展开「块设置」（非整页跳转） |
| **字段芯片** | 取数后列名以 Tag 展示，点击插入 `{{列}}` 到当前文本 |
| **主题条** | 顶栏：主题色 + 表风格（商务/紧凑/告警）全局生效 |

**不做**：`/` 命令全量百科（可二期加）；自由拖到任意 x,y。

## 5. 屏布局（工作台三栏）

```text
┌─ 顶栏：名称 | 场景徽章 | 主题 | 保存 | 试推 ─────────────────────┐
├──────────────┬────────────────────────┬─────────────────────────┤
│ 左：结构      │ 中：当前块编辑 / 数据    │ 右：终态预览              │
│              │                        │                         │
│ · 数据槽      │ 选中「表图块」时：        │ [ 图 ] [ MD ] [ 结构 ]   │
│   SQL 摘要    │  样式预设 / 显示列…     │                         │
│ · 模块列表    │ 选中「文本块」时：        │  手机宽或消息宽画板       │
│   ① 标题文案  │  多行 MD + 插值         │  接近钉钉实际观感         │
│   ② KPI       │ 未选中：数据预览表       │                         │
│   ③ 明细表图  │                        │  生成中 skeleton         │
│   ④ 脚注      │                        │                         │
│ [+ 模块]      │                        │ [刷新预览]               │
└──────────────┴────────────────────────┴─────────────────────────┘
```

| 栏 | 职责 | 宽度建议 |
|----|------|----------|
| 左 | 大纲 = 用户心智中的「这篇推送有几段」 | 240–280px |
| 中 | 当前上下文编辑（数据或块属性） | flex |
| 右 | **唯一真相预览**（最终载体，不是配置摘要） | 360–420px |

视觉气质对齐 [[05-UI视觉设计]]：中等密度、中性灰 + 主色、少动效。预览区可用浅灰底 + 白卡片模拟消息气泡。

## 6. 内容块类型（第一期收敛）

> 类型少而强，优于类型多而废。第一期 **4 种业务块 + 1 种系统块**。

| 块类型 | 用户理解 | 数据 | 渲染出口 | 美观来源 |
|--------|----------|------|----------|----------|
| **hero_text** 标题/说明 | 开头几句话 | 可选首行插值 | `text_md` | 排版层级（H1/正文） |
| **kpi_row** 指标条 | 3–6 个数字亮点 | 指定列或首行 | 图内区域或 MD 粗体行 | 模板 HTML 卡片样式 |
| **table_image** 数据表图 | 「好看的表」 | 绑定某数据槽 | `image`（HTML→PNG） | **主美观战场**：CSS 主题 + 条件色 |
| **footer_text** 脚注 | 口径说明、联系人 | 插值 | `text_md` | 小字灰色 |
| **divider_note**（可选） | 分段小标题 | 无 | `text_md` | 轻分隔 |

**二期再开**：

| 块 | 说明 |
|----|------|
| `file_export` | 结果导出 Excel/CSV 作为附件 part |
| `chart_image` | 简单柱/折（基于结果集） |
| `alert_banner` | 告警专用色条 + 级别 |
| `raw_markdown` | 高级用户整段 MD |

### 6.1 块配置原则（保持简单）

每个块设置 **≤ 7 个可见字段**；其余进「高级」。

示例 `table_image`：

- 数据槽（默认 main）  
- 表风格：商务 / 紧凑 / 告警  
- 条件着色：开/关（规则用全局或预设）  
- 最大行数  
- 显示表头：开/关  

**不在块上暴露**完整 CSS 编辑（管理员主题包另入口）。

## 7. 数据模型（草稿 Schema）

```json
{
  "version": 2,
  "scene_id": "daily_report",
  "theme": {
    "color": "#1677ff",
    "table_style": "business",
    "logo_url": null
  },
  "data_slots": [
    {
      "id": "main",
      "label": "主数据",
      "data_source_id": "…",
      "sql": "SELECT …"
    }
  ],
  "blocks": [
    {
      "id": "b1",
      "type": "hero_text",
      "visible": true,
      "config": {
        "title": "{{院区}} 运营日报",
        "body": "统计日期 {{stat_date}}"
      }
    },
    {
      "id": "b2",
      "type": "kpi_row",
      "visible": true,
      "config": {
        "slot": "main",
        "metrics": [
          { "column": "outpatient", "label": "门诊量" },
          { "column": "inpatient", "label": "住院" }
        ]
      }
    },
    {
      "id": "b3",
      "type": "table_image",
      "visible": true,
      "config": {
        "slot": "main",
        "style": "business",
        "color_ratios": true,
        "max_rows": 50
      }
    },
    {
      "id": "b4",
      "type": "footer_text",
      "visible": true,
      "config": { "body": "数据来源：数据中台 · 如有疑问联系…" }
    }
  ],
  "compose": {
    "mode": "image_primary",
    "include_markdown_caption": true
  }
}
```

| 字段 | 含义 |
|------|------|
| `data_slots` | 支持多查询；一期可只实现 `main`，结构先留扩展 |
| `blocks` | 有序数组；`visible` 支持「不做删除的软关」 |
| `compose.mode` | `image_primary` / `markdown_primary` / `mixed` — 控制 parts 合成策略 |

### 7.1 编译管线

```text
Draft
  → 执行 data_slots（并行/按依赖）
  → 对每个 visible block：block_renderer(type) → 中间片段
  → composer 按 compose.mode 合并为 Message.parts[]
  → 预览 API / 试推 / JobRun 共用同一条编译函数
```

**关键约束**：预览与正式运行 **同一 `compile_draft()`**，禁止两套逻辑。

### 7.2 兼容迁移

| 旧 | 新 |
|----|----|
| `design.output_mode=image` + template | 单 `table_image` 或 scene 展开的 blocks |
| `header_text` / `footer_text` | `hero_text` / `footer_text` |
| 无 design | 空白 scene + 用户自加块 |

读取时：`if version<2: design_to_draft_v2(design)`。

## 8. 美观策略（丰富内容从哪来）

### 8.1 分层

| 层级 | 手段 | 谁维护 |
|------|------|--------|
| **主题 Theme** | 主色、字体栈、表头底色、圆角、行高 | 系统 3–5 套 |
| **表风格** | business / compact / alert CSS | 系统 |
| **条件着色** | 百分比、阈值、阈值区间 | 块开关 + 全局规则引擎（二期细） |
| **场景版式** | 块默认组合 + KPI 栅格 | 场景模板 |
| **用户微调** | 文案、显隐、列、颜色覆盖 | 业务 |

**80% 美观来自 Theme + 表 CSS + 场景**，不是用户手调。

### 8.2 成图主路径（延续已定方向）

```text
blocks → HTML 片段拼装（有语义 class）→ 统一 chrome（页眉色条）
  → Playwright / wkhtml / Pillow 回退 → PNG
  → 可选附带 Markdown 摘要 part（Webhook 友好）
```

### 8.3 预览即终态

右侧预览必须是：

1. **消息宽**模拟（不是整页后台表格）  
2. 图与 MD Tab  
3. 「结构」Tab 显示将生成的 parts 列表（排障用，默认折叠）

## 9. 场景模板库（简单操控的第一入口）

| scene_id | 名称 | 默认块序列 | 适用 |
|----------|------|------------|------|
| `daily_report` | 院区/业务日报 | hero + kpi + table_image + footer | 最高频 |
| `alert` | 指标告警 | alert 文案 + table_image(alert 风格) | 异常推送 |
| `kpi_brief` | 指标简报 | hero + kpi_row（可无表） | 轻量 |
| `table_only` | 一张表 | table_image | 迁移过渡 |
| `blank` | 空白 | 无块，引导 +添加 | 高级 |

**创建交互**：

- 工作台空白态：大卡片选场景（主）  
- 「从模板重新应用」：二次确认是否覆盖当前块（保留 SQL）

## 10. 与通道 / 任务的边界（功能分配）

| 能力 | 内容工作台 | 任务管理 | 通道配置 |
|------|------------|----------|----------|
| SQL / 块 / 主题 / 预览 | ✅ | 入口跳转 | ❌ |
| 试推 | ✅ | 可选「执行」 | ❌ |
| 启用、Cron、最近运行 | 只读提示 | ✅ | ❌ |
| Webhook/OpenAPI 凭证 | 引用选择 | 引用 | ✅ 编辑 |
| 场景模板管理 | 使用 | — | 系统设置后期 |

延续 [[15-交互理念重设计]]：**工作台是创作主场，任务是运营，通道是投递适配。**

## 11. 分阶段落地（建议）

### Phase A — 骨架可感（约 1 个迭代）

- Draft schema v2 + 旧 design 兼容读  
- 前端三栏壳 + 模块列表（先固定场景展开的块，可显隐、排序）  
- 编译：hero + table_image + footer → Message  
- 预览与保存走 compile  

**验收**：选「日报」→ 改 SQL → 预览是「标题+表图+脚注」，不再是单表单。

### Phase B — 丰富与美观

- KPI 块  
- 表风格 3 套 + 条件着色打磨  
- 主题色全局  
- 字段芯片插入  

### Phase C — 编排与多样

- 任意增删块、多数据槽  
- 附件块  
- 自定义场景（存库）  
- 块级条件显示（行数为 0 则隐藏等）

### Phase D — 效率

- `/` 快捷插入  
- 版本/上次预览图  
- 从历史任务克隆

## 12. 技术落点（工程约束）

| 项 | 建议 |
|----|------|
| 前端 | 保持 React + Ant Design；块列表自研即可，**不**上完整 BlockNote/Notion 引擎（除非 Phase D 证明需要） |
| 后端 | `compile_draft(draft, query_results) -> Message` 单函数；块注册表 `block_type -> render` |
| 存储 | `push_jobs.render_spec = { "version": 2, "draft": {…} }` 或并列 `design` 过渡 |
| 测试 | 每个块类型 + 每个 scene 黄金快照（HTML 或 parts JSON） |
| 插件 | Renderer/Channel 不变；块是 editor 域概念，编译后仍是 Message parts |

## 13. 风险与取舍

| 风险 | 缓解 |
|------|------|
| 做成「小 Notion」工期爆炸 | 块类型上限；禁止画布；场景模板兜底 60% |
| 预览慢（成图） | 防抖、缓存上次 HTML、先出 MD 后出图 |
| 旧任务不兼容 | 读写兼容层，禁止强制迁移 |
| 美观仍不够 | 优先砸 CSS 主题与 HTML 表，不砸自由布局 |

---

## 14. 一页纸决策摘要

1. **心智**：竖向简报模块，不是表单，不是画布。  
2. **简单**：场景模板 + 渐进披露（填空 → 调块 → 编排）。  
3. **丰富**：多块组合 + HTML 表图主题 + 文图混排 compose。  
4. **美观**：系统主题/表风格承担 80%；用户少调参。  
5. **架构**：Draft → compile → Message；通道不进编辑器。  
6. **落地**：Phase A 先让「模块列表 + 终态预览」可感知，再堆块类型。
