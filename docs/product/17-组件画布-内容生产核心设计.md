# 组件画布 · 内容生产核心设计

> **用户方向**：做成画布；自定义组件 + 绑数据 → 组合成最终结果 → 生成图片或文案 → 再搭配通道；  
> 或像 **帆软** 那样：组件做好后搭配出最后效果。  
> **判定**：这是产品 **核心**，要深入设计，不是附属 UI。  
> 关联：[[16-内容工作台方案设计]]（前序模块思路）· [[13-内容生产管线-核心关注]] · [[15-交互理念重设计]]

## 0. 结论先行（架构取舍）

### 0.1 三种「画布」不要混谈

| 形态 | 代表 | 能力 | 对我们 |
|------|------|------|--------|
| **A. 无限自由画布** | Figma / 部分大屏 | 任意 x,y、缩放旋转、图层 | ❌ **否**。IM 推送宽度固定、无「读者缩放」；学习成本高、对不齐、难维护 |
| **B. 报表/决策画布** | **帆软决策报表**、部分邮件设计器 | **固定画板** + 组件拖放 + 数据集绑定 + 预览导出 | ✅ **主形态** |
| **C. 流式块文档** | Notion / 邮件竖向堆叠 | 只有上下顺序，无坐标 | ✅ 作为 B 的 **简易模式** / 默认布局策略 |

**我们要的不是 Figma，是「推送版帆软决策报表」：**

```text
数据集(SQL) ──绑定──► 组件实例 ──排布在──► 推送画板(固定宽)
                              │
                              ▼
                    编译导出：PNG 成图 和/或 文案(MD)
                              │
                              ▼
                    搭配 Channel（Message ⊥ 通道）
```

### 0.2 与文档 16 的关系

| 16 方案 | 17 深化 |
|---------|---------|
| 竖向模块列表 | 模块 = **画板上的组件实例**（可流式排布，也可半绝对） |
| 场景模板 | = **画板预设**（组件树 + 主题 + 默认数据集槽） |
| 三栏结构 | 升级为经典设计器：**组件库 \| 画板 \| 属性/数据** |

**16 不废，升维为画布的一种布局模式（流式）。**

## 1. 为什么核心必须是「组件 + 画板」

### 1.1 业务真相

推送「好看且多样」≈ 在一张 **固定宽度的视觉平面** 上，用 **可复用视觉单元** 拼版，并让每个单元 **吃数据**。

这和：

- 帆软：报表块/图表/文本 → 绑数据集 → 预览 → 导出  
- 旧系统：HTML 区块拼大页 → 截图  

是 **同一类问题**，不是「表单填几个字段」能覆盖的。

### 1.2 能力分解（核心能力地图）

```text
┌──────────────────────────────────────────────────────────┐
│                    内容生产核心                           │
│                                                          │
│  ① 数据集 Dataset     取数、参数、多结果集、预览表         │
│  ② 组件库 Component   类型、默认样式、数据契约             │
│  ③ 画板 Artboard      宽高策略、背景、主题、网格/辅助线     │
│  ④ 实例 Instance      组件落板、布局、绑定、覆盖样式        │
│  ⑤ 编译 Compiler      → 图片 / 文案 / 混合 Message         │
│  ⑥ 搭配 Delivery      选通道、试推（非画板内部逻辑）        │
└──────────────────────────────────────────────────────────┘
```

①–⑤ 是 **核心**；⑥ 是已有通道域，工作台只 **引用**。

## 2. 对标：帆软给我们什么 / 我们裁什么

### 2.1 帆软相关能力（决策报表 / 普通报表）

| 能力 | 说明 | 是否纳入一期核心 |
|------|------|------------------|
| 数据集（SQL） | 多数据集，组件各自绑定 | ✅ 必须 |
| 组件拖拽上画板 | 报表块、图表、文本、图片… | ✅ 必须（类型收敛） |
| 字段拖到组件/单元格 | 绑定数据列 | ✅ 表/KPI/文本插值 |
| 类 Excel 单元格报表 | 复杂中国式表 | ⚠️ **二期**；先 HTML 表组件顶住「好看表」 |
| 参数面板 | 查询条件 | ⚠️ 推送侧用「任务参数/运行时变量」后置 |
| 决策大屏自适应布局 | 盒子/绝对 | ✅ 用 **简化版**：流式 + 可选绝对 |
| 导出 PDF/Excel/图片 | 多出口 | ✅ 我们出口是 **PNG + MD(+file)** |
| 填报、权限、调度平台 | 全家桶 | ❌ 不做；调度在中台任务域 |

### 2.2 产品定义一句话

> **推送设计器** = 面向 IM 宽度的 **轻量决策报表画板**：  
> 组件库拖上画板 → 绑数据集 → 预览终态 → 一键成图/文案 → 搭配通道发出。

不是 BI 平台，不是 Excel 替代品。

## 3. 画板模型（Artboard）——「画布」的精确定义

### 3.1 画板不是无限空间

| 属性 | 约定 | 理由 |
|------|------|------|
| **宽度** | 固定档：`750`（默认）/ `600` / `1080`（大表） | 对齐手机/钉钉可读宽度；成图稳定 |
| **高度** | **内容撑开**（默认）或固定最大高度后分页/裁切策略 | 推送图通常一长条 |
| **坐标系** | 逻辑像素；导出时 scale=2 可视网膜 | 预览=导出一致 |
| **背景** | 色 / 简单渐变 / 无 | 主题 |
| **安全边距** | 左右 16–24px | 避免贴边 |

### 3.2 两种布局模式（同一套组件，两种排布）

| 模式 | 行为 | 谁用 | 实现难度 |
|------|------|------|----------|
| **Flow（流式）** | 组件自上而下自动堆叠，间距 theme.gap | **默认 80%** | 低 |
| **Free（半绝对）** | 在画板上拖 x,y；可对齐吸附；仍限制在画板宽内 | 标题叠图、左右分栏 KPI | 中 |

**原则：**

- 新建模板默认 **Flow**，简单。  
- 需要「左右 KPI + 下方大表」时：提供 **分栏容器组件**（Container），而不是鼓励满屏 Free。  
- Free 仅作为容器内或高级开关，避免一上来像素地狱。

```text
推荐结构（像帆软「布局容器 + 组件」）：

Artboard
  └─ VStack（根流式容器，隐式）
       ├─ Text / Hero
       ├─ HStack（分栏容器）
       │    ├─ Kpi
       │    └─ Kpi
       ├─ Table
       └─ Text footer
```

**容器组件** 是「简单操控」与「版式丰富」的平衡点——比纯绝对坐标好用，比纯单列更强。

## 4. 组件体系（Component System）——核心中的核心

### 4.1 组件 = 类型定义 + 实例

```text
ComponentType（定义，系统/可扩展）
  id, name, icon
  default_size / default_style
  data_contract   ← 需要什么数据形态
  prop_schema     ← 属性面板字段
  renderers       ← 如何画成 HTML 片段 / MD 片段

ComponentInstance（画板上的一个）
  id, type
  layout: { mode, x?, y?, w?, h?, order, parent_id? }
  props: { …用户覆盖 }
  binding: { dataset_id, …字段映射 }
  visible: bool | expression?
```

### 4.2 一期组件目录（宁少做透）

| 组件 | 数据契约 | 视觉 | 导出贡献 |
|------|----------|------|----------|
| **Text** | 无 或 单行/首行字段插值 | 标题/正文/脚注样式档 | MD 段落 + 图内文字 |
| **Table** | 二维表（dataset 全表或列裁剪） | HTML 主题表、条件色、斑马纹 | **主成图** |
| **Kpi** | 标量（首行一列 或 聚合） | 大数字+标签+环比色 | 图内卡片；MD 粗体行 |
| **Image** | 静态 URL 或 logo | 页眉 logo | 图内 |
| **Divider** | 无 | 线/空白 | 图内间距 |
| **Container** | 无（子组件） | VStack / HStack / Grid(2) | 布局 |

**二期组件**：Chart（柱/折）、AlertBanner、List、MetricCompare、FileExport（不进画板视觉，进 Message part）。

### 4.3 组件数据绑定（学帆软，简化）

帆软：数据集 → 拖字段到单元格/控件。  
我们：

| 绑定方式 | 适用 | UI |
|----------|------|-----|
| **整表绑定** | Table | 属性：数据集 = `ds_main`；可选列多选 |
| **字段绑定** | Kpi、Text 插值 | 下拉选列；或 `{{col}}` |
| **表达式**（后） | 可见性、阈值 | `row_count>0` |

```json
{
  "type": "Table",
  "binding": {
    "dataset_id": "main",
    "columns": ["院区", "门诊量", "同比"],
    "max_rows": 50
  },
  "props": {
    "style": "business",
    "color_ratios": true,
    "header_color": null
  }
}
```

```json
{
  "type": "Kpi",
  "binding": {
    "dataset_id": "main",
    "value_column": "门诊量",
    "label": "门诊量",
    "trend_column": "同比"
  }
}
```

### 4.4 组件渲染契约（双通道）

每个组件必须实现：

```text
render_html(instance, data_ctx) -> HtmlFragment
render_md(instance, data_ctx)   -> str | None   # 可空：纯装饰组件
```

画板编译：

```text
for each visible instance in tree order:
  html_parts.append(render_html(...))
  md_parts.append(render_md(...))  # skip None

full_html = wrap_chrome(theme, artboard, html_parts)
png = html_to_image(full_html)

message = compose(mode):
  image_primary: [image part] + optional caption md
  markdown_primary: [joined md]
  mixed: [md summary] + [image]
```

**同一组件树，两种投影**——这就是「生成图片 **或者** 文案，然后搭配」的工程含义。

## 5. 数据集（Dataset）——画板的数据层

### 5.1 模型

```json
{
  "datasets": [
    {
      "id": "main",
      "name": "主查询",
      "data_source_id": "…",
      "sql": "SELECT …",
      "params": {}
    },
    {
      "id": "summary",
      "name": "汇总",
      "data_source_id": "…",
      "sql": "SELECT …"
    }
  ]
}
```

| 能力 | 一期 | 二期 |
|------|------|------|
| 多数据集 | ✅ 结构支持，UI 至少 1–2 个 | 依赖、关联 |
| 运行预览 | ✅ 结果表 + 列芯片 | |
| 参数 | 简单 key-value | 日期控件等 |
| 缓存 | 预览会话内 | Redis |

### 5.2 与组件的解耦

- 数据集 **不** 知道组件。  
- 组件 **只** 通过 `dataset_id` + 字段名取数。  
- 换 SQL 不改版式；换组件不改 SQL —— **帆软同款心智**。

## 6. 工作台 UI（设计器布局）

学帆软/邮件设计器，而不是当前「单列表单」：

```text
┌─ 顶栏：推送名 | 画板宽度 | 主题 | 布局模式 | 编译预览 | 保存 | 试推 ─┐
├────────┬─────────────────────────────┬──────────────────────────────┤
│ 左侧   │ 中央                         │ 右侧                         │
│        │                             │                              │
│ 组件库 │     【 推送画板 】            │ Tab：属性 | 数据 | 结构        │
│ Text   │   ┌─────────────────┐       │                              │
│ Kpi    │   │  [Text 标题]     │       │ 选中组件 → 属性表单           │
│ Table  │   │  [Kpi][Kpi]     │       │ 绑定数据集/字段               │
│ …      │   │  [Table ……]     │       │                              │
│        │   │  [Footer]       │       │ 数据 Tab：SQL 编辑+运行       │
│ ────── │   └─────────────────┘       │ 列芯片点击绑定                │
│ 模板   │   可缩放 50%–100%            │                              │
│ 日报…  │   Flow 时只显示顺序与间距    │ 结构：组件树                  │
└────────┴─────────────────────────────┴──────────────────────────────┘
```

### 6.1 简单操控如何在「画布」上成立

| 技巧 | 做法 |
|------|------|
| **模板起步** | 打开先选「日报画板」已摆好组件，只改数据和文案 |
| **拖拽有约束** | 默认 Flow：拖的是 **顺序**，不是乱飞 |
| **容器优先** | 分栏用 HStack，不教用户对坐标 |
| **属性 ≤7** | 高级折叠 |
| **所见即所得预览** | 中央画板尽量接近成图；另开「导出预览」看 PNG/MD 真结果 |
| **双预览** | 设计态（可点选）+ 编译态（最终图/文） |

### 6.2 「搭配」放在哪

画板解决 **内容**；搭配是 **投递**：

| 步骤 | UI 位置 |
|------|---------|
| 成图 / 出文案 | 编译结果面板：Image / Markdown / Mixed |
| 选通道 | 顶栏或右侧「投递」：已有 Channel 多选 |
| 试推 | 顶栏主按钮 |
| 调度启用 | 仍在任务管理（可链过去） |

```text
[画板设计] → [编译出载体] → [搭配通道试推/保存任务]
     ▲ 核心 80%                ▲ 配置 20%
```

## 7. 端到端数据流（运行时）

```text
PushJob
  render_spec: {
    version: 3,
    artboard: { width: 750, theme_id, layout_default: "flow" },
    datasets: [...],
    tree: { root container + children instances },
    compose: { mode: "image_primary"|"markdown_primary"|"mixed" }
```

## 9. 成图与文案策略（搭配前的两种产物）

| compose.mode | 产物 | 适合 |
|--------------|------|------|
| `image_primary` | 整板 PNG + 可选短 MD 说明 | OpenAPI 发图；观感优先 |
| `markdown_primary` | 仅 MD（表用 MD 表或省略大表） | Webhook 文本 |
| `mixed` | MD 摘要 + 图 | 兼容性最好 |

**画板始终是视觉源真相**；MD 是投影。  
不要维护两套互不相干的排版。

### 10.3 前端画板实现策略（务实）

| 阶段 | 技术 | 说明 |
|------|------|------|
| Phase A | **DOM 流式画板**（不是真正 canvas 元素） | 用 HTML 排组件树；点选、排序；成本可控 |
| Phase B | 容器 HStack + 拖拽排序（dnd-kit） | 版式丰富 |
| Phase C | 可选 Free 布局（百分比/栅格吸附） | 高级 |
| 不选 | Fabric.js 无限画布一上来 | 过度 |

**「画布」是产品概念；一期实现可以是「可交互的 HTML 画板」，导出仍 HTML→PNG。**  
这与旧链路、现有 html_table 完全同路。

### 10.4 为什么不用真正的 `<canvas>` 绘图

| | HTML 画板 → 截图 | 2D Canvas 手绘 |
|--|------------------|----------------|
| 表/文字/CSS | 强 | 弱，自造排版引擎 |
| 与旧系统 | 一致 | 另起炉灶 |
| 条件着色 | CSS 即可 | 重写 |
| 维护 | 主题=CSS | 难 |

**导出用截图，编辑用 DOM** —— 工业界报表/邮件设计器的主流路径。

## 11. 与「简单」的矛盾如何解

画布天然更强也更难。强制规则：

1. **80% 流量走模板**：打开工作台 = 选画板模板，不是空白 Free。  
2. **默认 Flow**：用户感知更像「搭积木」而非「画图」。  
3. **组件数量控制**：一期 6 类；每类属性严格。  
4. **数据侧独立**：先跑通 SQL 出列，再点选绑定（降低认知并行）。  
5. **编译结果优先**：用户最终认 PNG/MD，不认设计器复杂度。  

若某交互让新手 10 分钟仍出不了预览图 → **判失败**，收回 Free/高级项。

## 12. 分阶段（核心路径重排）

### Phase S0 — 概念与契约（本设计）

- 锁定：Artboard + Component + Dataset + compile  
- 文档与 schema 草案  

### Phase S1 — 可拖的积木画板（核心 MVP）

- 组件：Text / Table / Kpi / Container(column)  
- 单数据集  
- DOM 流式画板 + 点选属性 + 绑定  
- compile → PNG + MD 预览  
- 1–2 个官方模板（日报、纯表）  

**验收**：拖/排组件 + 换 SQL → 出一张完整日报图。

### Phase S2 — 版式与美观

- HStack 分栏、主题包、表风格、条件色打磨  
- 双数据集  
- compose 三模式  

### Phase S3 — 设计器增强

- 拖拽排序/入容器  
- 组件显隐条件

## 13. 风险（核心域特有）

| 风险 | 等级 | 对策 |

## 15. 一页纸

1. **核心** = 数据集 + 组件库 + 固定宽画板 + 编译器。  
2. **画布** = 推送宽度的报表画板，不是 Figma。  
3. **学帆软** = 组件拖放、数据绑定、预览导出；不学填报/大屏全家桶。  
4. **默认 Flow + 容器分栏**，绝对定位为高级。  
5. **双投影** = 同一组件树 → HTML 成图 & MD 文案 → 再搭配通道。  
6. **实现** = DOM 画板 + HTML→PNG；组件插件化。  
7. **简单** = 模板起步 + 属性克制 + 组件少而精。
