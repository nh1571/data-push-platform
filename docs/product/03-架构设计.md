# 架构设计草案

> 依据：[[01-需求调研]]、[[02-领域模型]]。  
> 阶段：设计；不绑定具体目录树文件名（实现计划阶段再定）。  
> 技术栈方向：**Python**；Web API 倾向 FastAPI；异步执行倾向 Celery（可替换，接口以领域为准）。

---

## 1. 目标与非目标

### 1.1 目标（一期完整系统 A–F）

| 能力 | 架构对应 |
|------|----------|
| A 多源 SQL 取数 | DataSource 插件：MySQL、Doris |
| B 多形态渲染 | Renderer 插件：文本/MD、图片、卡片、文件 |
| C 通道投递 | Channel 插件：一期钉钉；消息⊥通道 |
| D 自建调度 | Scheduler 服务 + Schedule 配置 |
| E DS HTTP 触发 | 统一 Run API（与手工/定时同一入口） |
| F 日志与重跑 | JobRun / Delivery 持久化 + 新 JobRun 重跑 |

### 1.2 非目标（一期）

- 细粒度 RBAC（仅管理员登录预留）
- 企微等非钉钉通道实现（仅插件位）
- 非 SQL 取数
- 深度定制 DS 任务类型插件（HTTP 即可）
- 移动端 App

## 2. 逻辑架构

```text
┌─────────────────────────────────────────────────────────────┐
│                     管理端 / 调用方                            │
│   Web Admin  │  手工 Run  │  DolphinScheduler (HTTP)         │
└──────────────┴────────────┴───────────────────┬─────────────┘
                                                │
                     ┌──────────────────────────▼─────────────┐
                     │           API 层 (Application)          │
                     │  配置 CRUD  ·  统一 Trigger/Run  ·  查询 │
                     └──────────────────────────┬─────────────┘
                                                │
         ┌──────────────────────────────────────┼────────────────────────┐
         │                                      │                        │
         ▼                                      ▼                        ▼
┌─────────────────┐              ┌──────────────────────┐    ┌───────────────────┐
│  Config 上下文   │              │   Execution 引擎      │    │  Scheduler 上下文  │
│ DataSource      │              │  编排 JobRun 生命周期  │    │  Cron / 到期触发   │
│ PushJob         │              │  调插件 · 写状态/日志  │    │  → 调 Run 同一路径  │
│ Channel         │              └──────────┬───────────┘    └───────────────────┘
│ RenderSpec 等   │                         │
└─────────────────┘         ┌───────────────┼───────────────┐
                            ▼               ▼               ▼
                     DataSource       Renderer         Channel
                     Plugin SPI       Plugin SPI       Plugin SPI
                     mysql/doris      text/image/…     dingtalk
                            │               │               │
                            ▼               ▼               ▼
                         DB 集群         对象存储/本地临时    钉钉开放平台
```

**一句话**：所有触发（定时 / 手工 / DS / API）最终都进入 **同一条 Execution 流水线**；插件只出现在取数、渲染、投递三处。

## 3. 模块划分

| 模块 | 职责 | 不负责 |
|------|------|--------|
| **api** | HTTP：配置、触发、查询 JobRun、健康检查 | 不写具体 SQL 执行细节 |
| **config** | PushJob / DataSource / Channel / Schedule 的领域服务与仓储 | 不执行任务 |
| **execution** | JobRun 状态机、步骤编排、重跑创建新 Run | 不感知钉钉 API 细节 |
| **scheduler** | 扫描到期 Schedule、投递触发命令 | 不直接访问业务库做取数 |
| **plugins.datasource** | 连接与执行 QuerySpec | 不渲染、不推送 |
| **plugins.renderer** | QueryResult → MessagePart(s) | 不读业务库、不推送 |
| **plugins.channel** | Message + 通道配置 → 外部 IM | 不取数 |
| **identity** | 登录；一期全管理员 | 无复杂授权 |
| **common** | 密钥引用、日志、错误码、插件注册 | — |

模块依赖方向：**api → config/execution/scheduler**；**execution → plugins（仅接口）**；插件不依赖 api。

## 4. 核心运行时：Execution 流水线

### 4.1 统一入口

```text
TriggerRequest {
  push_job_id
  trigger_type: manual | schedule | external_http | rerun | …
  params: { biz_date?, … }
  requested_by?: operator | "ds" | "scheduler"
}
        │
        ▼
  创建 JobRun (pending)
        │
        ▼
  异步执行（Worker）→ running
        │
        ├─ 1. Load PushJob 配置快照（写入 JobRun 供审计；重跑用最新配置，O1）
        ├─ 2. DataSource.execute(QuerySpec, params) → QueryResult
        ├─ 3. 若「有数才推」且空结果 → succeeded + 无/skipped Delivery（T1）
        ├─ 4. Renderers → Message (MessagePart[])
        ├─ 5. 对每个 Channel：Delivery 尝试
        └─ 6. 汇总：succeeded | failed | partial（T3）
```

### 4.2 配置快照

JobRun 开始时固化 **PushJob 配置快照**（JSON），保证：

- 日志可解释「当时用的 SQL/通道」
- 重跑使用**最新配置**创建新 JobRun（O1 已确认）；快照仅审计

### 4.3 重跑（T2）

- API：`POST .../job-runs/{id}/rerun` 或 `POST .../push-jobs/{id}/run` 带 `rerun_of=id`
- 始终 **新 JobRun**，`parent_run_id` 可选外键指向原 Run

### 4.4 幂等与并发

| 场景 | 策略（草案） |
|------|----------------|
| DS 重复回调同一业务键 | 支持可选 `idempotency_key`；相同 key 在成功/运行中返回原 JobRun |
| 同一 PushJob 重叠调度 | **允许并行**（O3 已确认）；后续可再加「跳过/排队」策略开关 |
| Worker 崩溃 | JobRun 超时扫描置 failed 或 requeue（实现期定） |

## 5. 插件体系（SPI）

### 5.1 注册

- 进程启动时扫描/显式注册：`type_name → Plugin 实现`
- 配置里只存 `type` + `config JSON`（敏感字段加密或密钥引用）

### 5.2 DataSource Plugin

```text
interface DataSourcePlugin:
  type: str  # "mysql" | "doris"
  validate_config(config) -> None
  execute(config, query_spec, params) -> QueryResult
  # QueryResult: columns + rows（或流式游标接口二期）
```

一期实现：`mysql`、`doris`（协议可同为 MySQL 协议族，**仍分成两个 type** 以便差异配置/限流/方言）。

### 5.3 Renderer Plugin

```text
interface RendererPlugin:
  type: str  # "text_md" | "image_table" | "card" | "file_export"
  render(query_result, render_config, params) -> list[MessagePart]
```

| type（建议名） | 产出 MessagePart |
|----------------|------------------|
| text_md | kind=text，markdown/plain |
| image_* | kind=image，bytes/url |
| card | kind=card，结构化 JSON（通道可再适配） |
| file_export | kind=file，xlsx/csv 等 |

**卡片**：Renderer 产出**通道无关**的卡片模型；Channel 负责映射为钉钉 ActionCard 等（避免渲染层绑死钉钉）。

### 5.4 Channel Plugin

```text
interface ChannelPlugin:
  type: str  # "dingtalk"
  validate_config(config) -> None
  send(config, message: Message, delivery_target) -> DeliveryResult
```

- `DeliveryResult`：success / failure + provider_msg_id + raw_error
- 一期仅 `dingtalk`；config 内含应用凭证、默认 DeliveryTarget 等

### 5.5 扩展方式

新增库或 IM：实现对应 SPI + 注册，**不改** execution 主流程。

## 6. 调度与 DS 集成（混合 C）

```text
┌──────────────┐     TriggerRequest      ┌─────────────┐
│  Scheduler   │ ──────────────────────► │             │
└──────────────┘                         │  Run API    │
┌──────────────┐     HTTP POST           │  + Worker   │
│ DolphinSched │ ──────────────────────► │             │
└──────────────┘                         └─────────────┘
┌──────────────┐     手工 / 其他 API      │
│  Admin UI    │ ──────────────────────► │
└──────────────┘
```

| 入口 | 行为 |
|------|------|
| **Scheduler** | 读 Schedule，到期创建 TriggerRequest（type=schedule） |
| **DS** | 调用 `POST /api/v1/push-jobs/{id}/run`，Header 鉴权（Token），Body 带 biz_date 等 |
| **Admin** | 同 Run API，type=manual |
| **Rerun** | 同执行引擎，type=rerun |

**原则**：Scheduler 与 DS **都不**执行 SQL；只触发。

DS 侧配置建议：HTTP 任务 → 推送系统 Run URL；失败由 DS 重试策略 + 本系统 JobRun 状态双可见。

## 7. API 轮廓（应用层，非最终 OpenAPI）

| 领域 | 方法（示意） |
|------|----------------|
| 数据源 | CRUD `/data-sources`；`POST .../test` 测连 |
| 通道 | CRUD `/channels`；`POST .../test` 试发固定消息 |
| 推送任务 | CRUD `/push-jobs`；启用/停用 |
| 执行 | `POST /push-jobs/{id}/run`；`GET /job-runs`；`GET /job-runs/{id}`；`POST /job-runs/{id}/rerun` |
| 调度 | 随 PushJob 的 schedule 字段或子资源 |
| 鉴权 | 登录发 Token；Run API 支持 **人用户 Token** 与 **机器 Token**（给 DS） |

机器 Token 与人 Token 分离，便于后期权限分级。

## 8. 数据持久化（逻辑存储）

| 存储对象 | 说明 |
|----------|------|
| data_sources | 类型、配置（密文不落库：密钥 ID） |
| channels | 类型、配置、目标 |
| push_jobs | QuerySpec、RenderSpec、channel_ids、schedule、开关 |
| job_runs | 状态、触发信息、配置快照、时间、parent_run_id |
| deliveries | run_id、channel_id、状态、错误、外部 ID |
| job_run_logs | 步骤日志（取数耗时、渲染、投递） |
| operators / tokens | 一期管理员 + API Token |

**文件/图片**：一期 **本地卷 + 路径**（O4），库中存元数据；`Storage` 抽象便于日后换对象存储。

**数据库**：系统元数据使用 **PostgreSQL**（O2；与业务 Doris/MySQL 数据源分离）。

## 9. 部署视图（逻辑）

```text
[Web/API 进程] × N
[Worker 进程]  × M   ← 执行 JobRun
[Scheduler 进程] × 1（主备可后置）
[元数据库]
[Redis]（若用 Celery/队列；锁与 broker）
[文件存储]
```

一期最小：API + Worker + Scheduler + DB +（可选 Redis）。  
单机 docker-compose 可接受；与「完整系统」能力无关。

## 10. 安全（一期最小）

| 项 | 做法 |
|----|------|
| 数据源/通道密钥 | 加密存储或环境/密钥服务引用；API 回显脱敏 |
| Run API | 鉴权必选；DS 用机器 Token |
| 管理端 | HTTPS（部署层）；一期全管理员 |
| SQL | 仅配置侧可写 SQL，执行账号 **只读** 权限（运维约束，架构写进原则） |
| 审计 | JobRun/Operator 操作日志为 F 的一部分 |

## 12. Key Decisions

| # | 决策 | 理由 |
|---|------|------|
| K1 | 单一 Execution 流水线，多触发入口 | 满足混合调度 C，避免 DS/自建两套执行逻辑 |
| K2 | 三套插件 SPI：DataSource / Renderer / Channel | 对应用户强调的双插件化 + 渲染扩展 |
| K3 | 卡片中间模型与通道解耦 | 多通道时不重写渲染 |
| K4 | JobRun 配置快照 | 可观测、可追责 |
| K5 | 重跑 = 新 JobRun | 领域 T2 |
| K6 | 元数据与业务数仓分离 | 推送系统库 ≠ Doris/业务 MySQL |
| K7 | Python +（建议）FastAPI + 异步 Worker | 与已定技术栈及技能匹配；可替换实现不影响 SPI |
| K8 | 一期不做 RBAC 细节，预留 Operator + Token 种类 | 需求 Q8 |

## 13. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 大结果集 OOM | Query 限制行数/超时；文件导出流式写盘 |
| 图片渲染慢 | Worker 并发与超时；异步仅投递不阻塞 API |
| 钉钉限流 | Channel 内重试/退避；Delivery 级失败进 partial |
| DS 与本地调度双触发重复 | idempotency_key；文档约定调度主从 |
| 插件质量 | SPI 契约测试 + 每种插件最小集成测 |

## 14. 架构开放项结论（2026-07-16，O1–O5 均按默认）

| ID | 最终结论 |
|----|----------|
| **O1** | 重跑使用 **最新 PushJob 配置**；历史快照仅用于审计展示 |
| **O2** | 系统元数据库采用 **PostgreSQL** |
| **O3** | 同一 PushJob **允许并行** 执行多个 JobRun |
| **O4** | 大文件/图片一期：**本地卷 + 路径**，保留 Storage 抽象 |
| **O5** | 「有数才推」= PushJob 级布尔字段 **`skip_if_empty`** |
