# 数据推送中台（领域上下文）

数据中心将库内查询结果渲染为可投递消息，经通道插件送达企业内部 IM。本上下文只描述业务概念与统一语言，不绑定框架与表结构。

> 对应代码仓库落地时，可复制为根目录 `CONTEXT.md`。

## 统一语言（Language）

### 流水线核心

**推送任务（PushJob）**：
一条可重复执行的配置：指定如何取数、如何渲染、发到哪些通道。是运营配置的主对象。
_Avoid_：推送、任务（单独说时易与「执行」混淆）、作业配置

**执行实例（JobRun）**：
某次对推送任务的实际运行（含触发来源、参数如业务日期、起止状态与日志）。
_Avoid_：任务、运行记录（可作别名但不作主词）、instance 口语化

**取数结果（QueryResult）**：
一次执行中从数据源得到的结构化数据（行集/标量），尚未渲染。
_Avoid_：数据、结果集（可作描述）、Dataset（实现味）

**消息（Message）**：
已渲染、可投递的内容载荷（可含多种形态部件）。与「发给谁」无关。
_Avoid_：通知、推送内容、payload（实现味）

**消息部件（MessagePart）**：
消息中的一种形态单元：文本/Markdown、图片、卡片、文件等。一条消息可含多个部件。
_Avoid_：附件（仅指文件时用文件部件）、素材

### 数据源

**数据源（DataSource）**：
一个可连接的数据库端点配置（类型、地址、凭证引用等），供取数使用。
_Avoid_：连接、库、Connection（实现味）

**数据源类型（DataSourceType）**：
数据源的插件种类，如 `mysql`、`doris`。
_Avoid_：驱动、connector 名混用

**取数定义（QuerySpec）**：
推送任务上的「如何取数」：绑定数据源 + SQL（及参数占位等）。一期以 SQL 为主。
_Avoid_：SQL 任务、脚本（命令类后置再扩展）

### 渲染

**渲染器（Renderer）**：
将取数结果变为某类消息部件的能力（文本、图片、卡片、文件等）。
_Avoid_：模板引擎（过窄）、转换器

**渲染规格（RenderSpec）**：
推送任务上配置的「用哪些渲染器、模板/参数是什么」。
_Avoid_：模板配置（只覆盖一部分）

### 通道与接收

**通道（Channel）**：
一种投递插件类型的配置实例（如某个钉钉应用/机器人配置）。只负责把消息发出去。
_Avoid_：接收方（口语可说，主词用通道）、Webhook、IM

**通道类型（ChannelType）**：
通道插件种类，一期 `dingtalk`；后续 `wecom` 等。
_Avoid_：接收系统（描述性说法）

**投递目标（DeliveryTarget）**：
通道配置中的「发到哪里」（人、群等），属于通道侧配置，不是独立于消息的业务线。
_Avoid_：接收人列表作为与 Message 同级的一等公民；「机器人/应用」作为产品主分类

**投递（Delivery）**：
某次 JobRun 将某条 Message 交给某个 Channel 的一次尝试记录。
_Avoid_：发送、推送（与任务混淆）

### 调度与触发

**触发（Trigger）**：
引起一次 JobRun 的方式：定时、手工、条件、外部 HTTP（含 DS）、补跑等。
_Avoid_：调度（调度是机制；触发是一次起因）

**调度计划（Schedule）**：
推送任务上可选的本系统定时规则（Cron 等）。
_Avoid_：Cron（实现细节可作属性）

**外部触发（ExternalTrigger）**：
经 HTTP 等由系统外（如 DolphinScheduler）发起的触发；与自建调度并列入口。
_Avoid_：回调（方向易混）

### 身份（一期简化）

**操作者（Operator）**：
使用管理后台的人。一期均为管理员。
_Avoid_：用户角色细分（后置）

## 聚合与关系（概念）

```text
PushJob
  ├── QuerySpec ──► DataSource (DataSourceType 插件)
  ├── RenderSpec ──► 一个或多个 Renderer 配置
  ├── Channel 引用（1..n）── 各含 DeliveryTarget 等通道配置
  └── Schedule?（可选，本系统定时）

JobRun（属于某个 PushJob）
  ├── Trigger（来源 + 参数，如 biz_date）
  ├── QueryResult?
  ├── Message?（含 MessagePart[]）
  ├── Delivery[]（每个 Channel 一次或多次尝试）
  └── 状态 + 日志
```

**不变量（草案）**

1. Message 不包含「通道密钥」；只包含可投递内容。  
2. 同一 JobRun 可对多个 Channel 产生多个 Delivery。  
3. JobRun 失败可重跑：产生新的 JobRun 或对失败 Delivery 重试——见状态机待确认点。  
4. DataSource 与 Channel 均通过类型插件扩展，核心不感知 Doris/钉钉细节。

## JobRun 状态机（草案）

```text
          ┌──────────┐
          │  pending │  已创建，未开始
          └────┬─────┘
               │ 开始执行
          ┌────▼─────┐
          │ running  │  取数 / 渲染 / 投递中
          └────┬─────┘
       ┌───────┼───────┐
       │       │       │
       ▼       ▼       ▼
  succeeded  failed  partial
               │
               │ 重跑（策略待定）
               ▼
            (新 JobRun 或 retry)
```

| 状态 | 含义 |
|------|------|
| `pending` | 已接受触发，排队或未开跑 |
| `running` | 执行中 |
| `succeeded` | 取数+渲染+全部应投递通道成功 |
| `failed` | 关键步骤失败，未形成有效完整投递 |
| `partial` | 部分通道成功、部分失败（若允许多通道） |

**已确认（2026-07-16）**：需要 `partial`；重跑采用**新 JobRun**（见 T2/T3）。

## 关键场景（验收语言）

| 场景 | 过程 |
|------|------|
| 定时日报 | Schedule 到期 → Trigger=schedule → JobRun → SQL on Doris → 图+文本 Message → DingTalk Channel → Delivery succeeded |
| 手工试跑 | Operator 点执行 → Trigger=manual → 同上（可用限流/标记试跑） |
| 有数才推 | 取数后行数为 0 → JobRun=`succeeded`，无 Delivery 或 Delivery=`skipped`（T1） |
| DS 触发 | DS HTTP → ExternalTrigger → 同一 PushJob 的 JobRun |
| 多形态 | 一表数据 → 多个 MessagePart（MD + 图片 + 附件）→ 一次或分次 Delivery（由通道能力决定） |
| 失败重跑 | JobRun failed/partial → 重跑产生**新 JobRun**（T2） |

## 已拍板术语决策（2026-07-16，T1–T4 均按默认）

| # | 决策 |
|---|------|
| **T1** | 无数据不推：JobRun 为 **`succeeded`**；无 Delivery 或 Delivery=`skipped`（**不**设 JobRun 级 `skipped`） |
| **T2** | 失败/部分失败重跑：始终创建 **新 JobRun**（保留历史，类 DS 补跑） |
| **T3** | 多通道部分成功：JobRun 状态 **`partial`** |
| **T4** | 一期取数 **仅 SQL**（QuerySpec）；非 SQL/命令类后置，插件接口可预留 |
