# 内容生产管线：日常开发的主战场

## 1. 用户/业务方原话要点

- **旧思路终点**：先得到 **富文本（Markdown）**，再 **统一交给钉钉推送**。  
- 生成富文本过程中，用了 **自由度较高的生图、排版**（HTML+CSS 表、截图等）。  
- 能力够用，但 **受系统设计拖累，整套流程很麻烦**。  
- **数据获取 → 最终消息载体生成** 这一段 **极其重要**。  
- 系统日常 **大部分开发** 发生在这里。  
- 要重点抓：**简化操作** + **丰富样式**。

## 2. 产品分层（重新标定重心）

```text
┌─────────────────────────────────────────────┐
│  ★ 内容生产工作台（日常 80% 工作量发生在此）    │
│  取数 → 看数据 → 排版/生图/富文本 → 预览载体   │
└──────────────────┬──────────────────────────┘
                   │ 产出：Message（统一载体）
                   ▼
┌─────────────────────────────────────────────┐
│  投递层（相对稳定、配置化）                     │
│  钉钉 Mode（单发/群发/Webhook/工作通知…）      │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  调度/运维（定时、DS、日志、重跑）               │
└─────────────────────────────────────────────┘
```

| 层 | 变化频率 | 谁用 | 产品策略 |
|----|----------|------|----------|
| **内容生产** | 每天、每任务都改 | 数据/业务开发 | **极致降门槛、多模板、可预览** |
| 通道 | 少改 | 运维/管理员 | 配一次、Mode 清晰 |
| 调度/日志 | 少改 | 运维 | 可靠即可 |

**旧系统麻烦的根因**：把「内容生产」做成了 **写 Python + 拷样板 + 改 crontab**，而不是工作台。  
**新系统成败关键**：**内容生产是否好用**，不是通道表单字段多几个。

## 3. 旧链路 vs 新目标

### 旧（能力对、体验差）

```text
手写 getdata.py + md.py + chats.py + main.py
  → DataFrame → HTML/CSS → 截图 → 拼 Markdown
  → 工厂选 single/orggroup/webhook → 发出
```

- 富文本是「汇合点」  
- 生图/排版自由度高，但 **全是代码**  
- 改一个样式/加一列 = 改文件 + 上线流程重  

### 新（能力对齐、体验产品化）

```text
工作台：选数据源 + 写/选 SQL
  → 一键取数预览（表）
  → 选模板 / 调样式 / 绑定字段 / 可选生图
  → 实时预览「最终载体」（MD 和/或 PNG）
  → 保存为任务 + 试推
  → 通道只消费统一 Message
```

终点仍可以是：

- **主载体 A：Markdown 富文本**（快、兼容 Webhook）  
- **主载体 B：渲染图 PNG**（观感稳定、对齐旧「好看表」）  
- 或 **MD + 图** 组合  

通道层 **统一吃 Message**，不再让每个任务自己拼 SDK。

## 4. 「简化操作」要拆成可交付能力

| 目标 | 具体能力 |
|------|----------|
| 少写代码 | SQL + 模板参数，不写 .py |
| 少步骤 | 同屏：数据 | 设计 | 预览 | 试推 |
| 可复用 | 模板库（日报表/告警/KPI…），院区只换数据源与参数 |
| 可试错 | 不发钉钉也能预览；试推一键 |
| 可回看 | 任务版本 / 上次预览图（后置） |

## 5. 「丰富样式」要拆成可交付能力

| 层级 | 内容 |
|------|------|
| L1 文本 | Markdown、标题、插值 `{{列}}`、自动表 |
| L2 表图 | 表格主题 CSS、条件着色（对齐旧百分比红绿）、截图成图 |
| L3 版式模板 | 页眉 Logo、主题色、多区块（标题/指标/表/脚注） |
| L4 高级 | 图表（趋势）、图标库、画布拖拽（更后） |

**旧系统 L2 很强（HTML+CSS+wkhtml）——这是样式主粮，应作为新系统中期主路径，而不是只靠 Pillow 简表。**

## 6. 工程含义（给实现）

1. **编辑器 = 一等公民产品**，不是任务表单的附属 Tab。  
2. **Message 统一出口**：`parts[]` = text | image | file | card。  
3. **设计态 Design** 与 **渲染态 Render** 分离：Design 可存模板 id + 参数；Render 出最终 parts。  
4. **样式资产** 独立目录（类旧 `css/`）：主题可迭代不改业务任务。  
5. 通道只适配；**禁止**每个 Mode 复制一套排版逻辑。  

## 7. 与 HANDOFF 优先级的关系

内容生产管线 > 通道堆 Mode > 调度花活。

P0 应持续倾斜：

- 预览闭环（数据 + MD + 图）  
- 模板库与样式  
- 发图通道（OpenAPI 单/群）  
- SQL Server 等取数（支撑真实任务迁移）  

## 8. 变更

- 2026-07-16：用户明确「取数→载体」是日常开发主战场；简化操作 + 丰富样式为核心关注
