# 产品分析修正（第二轮体验反馈）

## 用户三点反馈

1. **通道**：钉钉不止 Webhook + 工作通知两种。  
2. **推送编辑**：不能只编 Markdown；要图标/图片等，**好看的模板渲染成图再发出**。  
3. **推送任务「新建」直接跳编辑**：是否不合理 / 功能分析不足。

---

## 一、钉钉「推送」能力全景（对照开放平台）

钉钉文档里，和企业「往外推消息」相关的，至少要分三层：

### 1）发送场景（投递形态）— 应做成 Channel Mode

| 场景 | 说明 | 典型 API/方式 | 数据中心用途 |
|------|------|----------------|--------------|
| **自定义群机器人 Webhook** | 固定群，URL 直推 | Webhook POST | 运维/播报群，最快接入 |
| **应用/群机器人 · 接口发送** | 推荐；支持图片等更全 | OpenAPI 机器人发群聊 | 正式群推，要图/文件时用这个 |
| **机器人 · 单聊** | 人与机器人会话 | 单聊机器人 API | 点对点机器人消息 |
| **工作通知** | 应用名义进「工作通知」 | `asyncsend` / corpconversation | 到人/部门的正式通知 |
| **场景群 / 群助手 / 群模板机器人** | 业务群模板体系 | 群助手消息、模板机器人 | 有场景群的业务线 |
| **互动卡片 / 卡片平台** | 可更新、可交互的卡片 | 卡片实例创建与投放 | 复杂交互（可后置） |
| **（少用）普通消息/员工代发** | 个人操作场景 | 另一套能力 | 数据推送中台一般不优先 |

机器人消息类型（接口 vs Webhook）也不对等，例如：

| 消息类型 | 接口机器人 | Webhook |
|----------|------------|---------|
| text / markdown / ActionCard / link | ✅ | ✅（FeedCard 等另有差异） |
| **图片 Image** | ✅ | ❌（Webhook 通常不能发真图） |
| 语音等 | ✅ | ❌ |

**含义：**  
若产品要把「表格/模板渲染成**图片**再发」，**不能依赖 Webhook  alone**，必须规划 **OpenAPI 机器人（或工作通知图片类型）** 作为主投递能力之一。

### 2）消息内容类型 — 应做成 Message 部件，而不是和 Mode 绑死

text、markdown、link、ActionCard、FeedCard、image、file、互动卡片…

### 3）产品里正确的两级选择

```text
通道配置：
  平台 = 钉钉
  方式 = Webhook | 机器人OpenAPI群 | 机器人单聊 | 工作通知 | 场景群助手 | …
  凭证与目标 = 随方式变

推送内容：
  与方式解耦；投递时由 Channel 适配成该方式支持的 payload
  若方式不支持图片 → UI 明确禁用或降级为链接
```

**当前不足：** 只暴露 2 个 mode，且产品文案像「钉钉只有这两种」——分析与信息架构都偏窄。

**修正后的 Mode 清单（产品应展示，实现可分期）：**

| mode 代码 | 名称 | 一期 | 二期 |
|-----------|------|------|------|
| `dingtalk.webhook_robot` | 群自定义机器人 Webhook | ✅ 已有 | |
| `dingtalk.work_notice` | 工作通知 | ✅ 已有壳 | 完善选人/图片 |
| `dingtalk.openapi_group_robot` | 应用机器人·发群（OpenAPI） | 🔲 **优先补** | |
| `dingtalk.openapi_oto_robot` | 应用机器人·单聊 | 🔲 | ✅ |
| `dingtalk.scene_group_helper` | 场景群/群助手 | 🔲 | ✅ |
| `dingtalk.interactive_card` | 互动卡片 | 🔲 | 可选 |

## 二、推送编辑：从「写 Markdown」到「模板成图」

### 用户要的

- 不只 markdown 文本  
- 能编 **图标、图片、版式**  
- **好看的模板** → **渲染成一张图** → 发出去（钉钉会话里观感稳定）

### 产品能力拆分

| 能力 | 说明 |
|------|------|
| **数据层** | SQL 取数 + 表格预览/刷新（已有雏形） |
| **设计层 MessageDesign** | 选模板（日报/告警/KPI 卡…）；绑定字段；标题/指标/表格区/页脚；主题色/Logo |
| **渲染层** | 设计 + 数据 → **PNG/JPG**（主交付）+ 可选附带 markdown 摘要 |
| **投递层** | 优先走 **支持图片的通道**（OpenAPI 机器人 / 工作通知 image） |

### 编辑器应有的区块（目标交互）

```text
[取数] SQL + 运行 + 结果表
[模板] 选模板 / 空白画布
[画布] 可视化：标题、指标块、表格、图标、Logo、背景
[绑定] 字段拖到组件上
[预览] 实时出图预览
[通道] 选择支持图片的 mode
[试推/保存]
```

当前「header + footer + 是否附表」只是 **最小文本方案**，与目标差距大——反馈正确。

### 实现分期

| 阶段 | 交付 |
|------|------|
| 2a | 内置 2～3 套 **固定 HTML/模板 → 截图成图**（Playwright/wkhtml 或 PIL 精修模板） |
| 2b | 模板参数：标题、Logo URL、主题色、显示哪些列 |
| 2c | 画布式拖拽（后置） |

## 三、「推送任务」新建跳编辑 —— 是设计问题，不是小 bug

### 现在怎么做的

- 「推送任务 → 新建」→ 重定向到 **推送编辑**  
- 原因：当时把「编辑工作台」当成唯一入口，任务列表当「已保存方案列表」

### 为什么你会觉得别扭

| 问题 | 说明 |
|------|------|
| **两个菜单语义打架** | 「任务」像运维配置列表；「编辑」像创作工具；新建却混在一起 |
| **任务不止「内容」** | 任务还有调度、启用、跳过空数据、多通道、权限——编辑器侧重内容 |
| **缺少「任务实体」心智** | 用户期望：先建任务壳（名称/分类），再进编辑；或从编辑保存后出现在任务列表 |
| **分析不足** | 阶段 1 把「主路径」做成跳转，没有把 **对象模型（任务 vs 草稿 vs 编辑会话）** 讲清楚 |

### 更合理的信息架构（修正）

```text
【推送任务】—— 方案/作业的管理中心（列表、状态、调度、最近运行）
   ├─ 新建任务 → 弹窗：名称 + 可选模板 → 创建空任务 → 进入编辑器
   ├─ 编辑内容 → 打开编辑器（带 jobId）
   ├─ 运行 / 调度 / 复制 / 停用
   └─ 不在列表上「假装只有编辑这一个动作」

【推送编辑】—— 创作工作台（可从任务进入，也可「从草稿开始」）
   ├─ 从任务进入：URL /editor/:jobId
   ├─ 从空白进入：先提示「保存时创建任务」或「先选/建任务」
   └─ 不与「任务列表的新建」用同一种含糊跳转

【通道 / 数据源】—— 资源库
```

**新建任务推荐交互：**

1. 点「新建任务」→ Modal：任务名称、可选业务分类  
2. 创建成功 → **自动进入** 推送编辑（此时跳转合理，因为已有任务实体）  
3. 或：创建后留在列表，点「编辑内容」再进  

**不要：** 点「推送任务-新建」却静默变成「编辑器且还没有任务 id」（当前体验）。

## 四、结论：哪些是分析不足

| 点 | 判定 |
|----|------|
| 钉钉只有两种 | **能力分析不足**（开放平台场景远多于 2） |
| 编辑只有 markdown | **产品主路径定义过窄**（用户要的是「成图模板」） |
| 任务新建=跳编辑 | **信息架构不清晰**（任务管理 vs 内容创作未拆干净） |

3. **修正导航与新建任务流程**
