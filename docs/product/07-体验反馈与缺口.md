# 体验反馈与严重缺口（2026-07-16）

## 用户原话摘要

1. **通道只支持大类（如「钉钉」）**，但钉钉开放平台并非只有 Webhook 一种方式。  
2. **缺少推送编辑（最重要）**：需要完整的数据编辑能力——从写入取数代码（SQL）开始，**可视化**拿到数据，再编辑成想要的结果效果，然后推送。

## 对照：当前实现实际是什么

| 环节 | 现状 | 用户期望 |
|------|------|----------|
| 通道 | 类型=`dingtalk`，表单实质是 **Webhook URL + 简单 msgtype** | 钉钉下多种投递能力可选、可配 |
| 任务 | Tabs：写 SQL、勾渲染类型、选通道、点执行 | **编辑器工作台**：取数→预览数据→编排效果→预览消息→推送 |
| 渲染 | 勾选 text_md / image_table / card / file_export，配置很薄 | 可视化把「表数据」变成「最终消息效果」 |
| 试跑 | 整任务执行到钉钉 | 中间步骤可预览、可改、可再推 |

**结论：** 当前版本是「配置 CRUD + 管道执行」的骨架，**不是**「内容生产与编辑平台」。用户判定的「功能很简单 + 严重缺失」成立。

---

## 缺口 1：通道应分层（平台 → 方式 → 凭证/目标）

钉钉开放能力（不完全列举，以开放平台为准）：

| 方式 | 典型场景 | 与 Webhook 差异 |
|------|----------|-----------------|
| **群自定义机器人 Webhook** | 往固定群丢消息 | 配置简单；消息类型有限；无应用身份 |
| **机器人 OpenAPI（接口发）** | 应用机器人发群聊/单聊，图文卡片更全 | 需 access_token / open_conversation 等 |
| **工作通知（corpconversation）** | 以应用名义推到员工工作通知 | 需 userid/部门；有频控与人数限制 |
| **其他**（群消息、卡片互动等） | 更重的企业应用场景 | 凭证与 API 不同 |

**模型修正（替代「只有一个 dingtalk」）：**

```text
ChannelProvider = dingtalk | wecom | …
ChannelMode     = webhook_robot | openapi_robot | work_notice | …
Channel         = provider + mode + config(凭证/目标)
```

- UI：先选 **平台（钉钉）**，再选 **发送方式**，表单按 mode **动态字段**。  
- 插件：`ChannelPlugin.type` 建议改为 `provider.mode`（如 `dingtalk.webhook_robot`），或 provider 插件内多 mode。  
- 一期至少落地：**Webhook 机器人 + 工作通知**（或 Webhook + OpenAPI 机器人）二选一由用户定优先级。

消息与通道仍解耦：**编辑器产出 Message**；**Channel 只负责按 mode 投递**。

## 缺口 2：推送编辑工作台（产品核心，非附属 Tab）

用户要的不是「任务表单里填 SQL + 勾渲染」，而是：

```text
① 写/管理 SQL（或取数代码）
        ↓ 试运行
② 可视化查看取数结果（表格，可抽样、可刷新）
        ↓
③ 编辑「如何变成推送效果」
   - 选/编排部件：文本、表格图、卡片、附件…
   - 映射列、模板、标题、过滤/聚合（按能力分期）
   - 实时/准实时 预览最终消息长什么样
        ↓
④ 选择通道（含 mode）与目标
        ↓
⑤ 试推 / 正式推 / 保存为推送任务
```

**建议一等公民概念：**

| 概念 | 含义 |
|------|------|
| **PushDraft / 推送方案** | 一次可编辑的方案（可存为 PushJob） |
| **QuerySpec** | 取数定义（保留） |
| **DataPreview** | 取数结果预览会话（可缓存） |
| **MessageDesign** | 可视化编辑的「效果设计」（替代薄弱的 render_spec 勾选） |
| **MessagePreview** | 按设计 + 当前数据渲染出的预览 |
| **ChannelBinding** | 绑定哪些通道/mode |

**UI 重心：** 独立「推送编辑」主路径（全屏或主导航一级），列表任务是「已保存方案」；通道/数据源是资源库。

## 与旧设计的关系

| 仍保留 | 必须升级 |
|--------|----------|
| 数据源插件、执行实例、日志重跑、调度/DS 触发 | 通道 mode 分层 |
| Message ⊥ Channel 原则 | **Message 如何被「编辑出来」成为核心产品** |
| 插件化扩展 | 渲染从「勾类型」升级为 **可视化 MessageDesign** |

旧版「任务 Tabs」可降为高级/运维视图，**不能**再当主交互。

## 建议的迭代优先级（待用户确认后开发）

| 优先级 | 内容 |
|--------|------|
| **P0** | 推送编辑工作台：SQL → 跑数预览表 → 简单效果编辑（至少 MD 模板+列插入）→ 消息预览 → 试推 |
| **P0** | 通道：钉钉下 **多 mode** 选择 + 动态表单（Webhook 保留；+ 至少再一种：工作通知 或 OpenAPI 机器人） |
| **P1** | 可视化表格图/卡片设计器增强 |
| **P1** | 工作通知选人/部门 |
| **P2** | 企微等多平台 mode |
