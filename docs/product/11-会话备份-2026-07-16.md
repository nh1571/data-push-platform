## 2. 产品结论（必须记住）

### 2.1 定位

- **数据中心推送中台**：SQL 取数 → 渲染/编辑效果 → 通道投递  
- 场景：数据播报 + 配置化「SQL/命令有问题就推」（同一流水线）  
- **非目标**：FCM/APNs、C 端 App 推送  

### 2.2 核心原则

| 原则 | 内容 |
|------|------|
| Message ⊥ Channel | 消息生成与接收方独立；接收方 = 通道配置 |
| 双插件 | DataSource + Channel（Renderer 也是插件） |
| 一期通道平台 | 钉钉为主；架构插件化可扩企微 |
| 数据源一期 | **Doris + MySQL** |
| 渲染目标 | 文本/MD、图片、卡片、文件 **均要** |
| 调度 | **混合 C**：自建调度 + DS **HTTP** 触发；执行始终在中台 |
| 权限 | 一期全管理员；RBAC 后置 |
| 成功标准 | A–F 全要才算完整系统 |
| 元库 | **MySQL 8**（系统配置/日志，不是业务 Doris） |

### 2.3 用户关键纠正

1. **不要**把钉钉拆成「机器人 vs 应用」两套业务产品线；通道是配置维度。  
2. 又补充：**钉钉开放能力远不止两种 mode**；发图要靠 OpenAPI 等，Webhook 弱。  
3. **主产品是编辑**：看见数据 → 编效果（含图标/模板成图）→ 再推。  
4. 「推送任务新建直接跳编辑」**不合理**，任务管理 vs 内容创作要拆清。
