## 核心模型（用户纠正后，2026-07-16）

> **消息与接收方独立。** 接收方 = 不同推送通道的配置；消息生成完成后交给通道即可。  
> 不在业务上拆成「机器人场景 / 应用消息场景」两套产品逻辑。

```text
[数据源插件]  --SQL等-->  [取数结果]
                              |
                              v
                        [渲染：图片 / 富文本 / …]
                              |
                              v
                        [消息 Message]
                              |
                              v
                        [通道插件 Channel]  ← 仅配置不同（一期钉钉）
                              |
                              v
                           人 / 群 / …（通道侧配置）
```

| 层 | 职责 | 扩展方式 |
|----|------|----------|
| **数据源** | 连接多种数据库，执行 SQL 等取数 | **插件化**；先接几种，统一接口后续扩 |
| **消息** | 取数结果渲染为图片、富文本等 | 渲染类型可扩展 |
| **通道** | 把已生成消息发出去 | **插件化**；一期钉钉，后续企微等 |

### 废弃/纠正的理解

- ~~按钉钉「群机器人 vs 应用消息 vs 群会话」做产品主分类~~ → 过细、过早；通道内部实现细节，不升为业务模型主轴
- 告警与播报仍可同属「取数 → 渲染 → 推送」，不必拆两套系统

## 结论区

### Q1 业务场景

- **A + B**：数据播报 + 配置化问题推送（预写 SQL/命令，有问题/有结果再推）
- 本质统一为数据推送流水线

### 架构原则（强约束）

1. **Message ⊥ Channel**：消息生成与投递解耦  
2. **Channel 插件化**：一期只实现钉钉；接收方 = 通道配置  
3. **DataSource 插件化**：多数据库可连；统一接口扩展  
4. **取数方式**：以写 SQL 等为主从库取数  
5. **渲染**：结果 → 图片、富文本等再进入 Message  

### 通道策略

| 项 | 结论 |
|----|------|
| 一期 | 仅钉钉 |
| 扩展 | 插件接入其他接收系统 |
| 到人/到群 | 作为**通道配置项**，不是独立产品线 |

### Q4 一期数据源（2026-07-16）

| 数据库 | 优先级 |
|--------|--------|
| **Doris** | 一期必须 |
| **MySQL** | 一期必须 |
| 其他 | 二期起按统一 DataSource 插件接口扩展 |

### Q5 消息渲染（2026-07-16）

**以下均为必须支持（非二期可砍）：**

| 形态 | 说明 |
|------|------|
| **A. 纯文本 / Markdown 富文本** | 段落、列表、简单表格文字等 |
| **B. 图片** | 表格/图表等渲染为图再发 |
| **C. 结构化卡片** | 如钉钉 ActionCard 等（标题、按钮、跳转等） |
| **D. 文件附件** | 如 Excel/CSV 等明细投递 |

渲染层应可组合或按任务配置选用；与通道解耦——通道负责投递已生成的各形态载荷。

### Q6 触发方式（2026-07-16）

**全部需要**，能力对齐 **DolphinScheduler 类调度/触发**，而非只做简单 Cron。

| 类型 | 需求 |
|------|------|
| 定时调度 | Cron 等周期执行 |
| 手工执行 | 立即跑 / 试跑 |
| 条件触发 | 有数据/阈值等条件满足才推（或作为任务逻辑一部分） |
| 外部 API | 他系统触发本系统任务 |
| DS 类扩展（设计 implicit） | 补数/重跑、依赖/工作流编排、失败重试等——实现时可对照 DS，是否一期全做待拆 |

> 注：是「触发能力像 DS」，还是「直接对接/嵌入 DS」——已结论为混合，见 Q7。

### Q7 调度落地（2026-07-16）

**结论：要 B，并附带 A1 → 即混合模式 C。**

| 部分 | 内容 |
|------|------|
| **B 自建调度** | 系统内具备类 DS 的触发能力：定时、手工、条件、补跑等（独立可跑） |
| **A1 对接 DS** | 提供 **HTTP API** 供现有 DolphinScheduler 触发「执行某推送任务」 |
| **执行权** | 始终在推送系统（SQL→渲染→通道）；DS/自建调度只负责「何时触发」 |
| **等价表述** | 用户所述 B+A1 **就是 C（混合）** |

```text
[本系统调度] ──┐
               ├──► Run API / 执行引擎 ──► 取数→渲染→通道
[DS HTTP 任务] ─┘
[手工/API 调用] ─┘
```

### Q8 使用者与权限（2026-07-16）

| 项 | 结论 |
|----|------|
| **一期** | **D：统一管理员**，不做细粒度角色分级 |
| **后期** | 会补权限分级；**方案未定**，设计时预留扩展点即可，不在一期落地 RBAC 细节 |
| **设计含义** | 登录/鉴权可先做「能登录的都是管理员」；权限模型表结构或接口勿写死成无法扩展 |

### Q9 一期成功标准（2026-07-16）

用户判断：**A–F 都要，才算完整系统**（非可砍 MVP 子集）。

| 编号 | 必须具备 |
|------|----------|
| **A** | 可配置 Doris / MySQL 数据源，写 SQL 取数 |
| **B** | 可渲染并投递：文本/MD、图片、结构化卡片、文件附件 |
| **C** | 可配置推送通道（一期钉钉）；消息与通道分离 |
| **D** | 本系统自建调度：定时 + 手工等 |
| **E** | DolphinScheduler 可通过 **HTTP** 触发同一执行链路 |
| **F** | 执行日志：成功/失败可查、可重跑 |

> 一期范围 = 上述完整闭环，而不是「先只做发消息 demo」。

## 需求调研摘要（P0 收敛稿）

### 一句话

> 数据中心推送中台：多源 SQL 取数 → 多形态渲染 → 通道插件投递（一期钉钉）；调度自建并支持 DS HTTP 触发；消息与接收方解耦。

### 流水线

```text
DataSource 插件(Doris/MySQL)
    → SQL 取数
    → Render（文本/MD | 图 | 卡片 | 附件）
    → Message
    → Channel 插件（一期钉钉）
```

### 约束清单

| 维度 | 结论 |
|------|------|
| 场景 | 数据播报 + 配置化问题推送（统一流水线） |
| 数据源 | 插件化；一期 Doris + MySQL |
| 渲染 | A/B/C/D 均必须 |
| 通道 | 插件化；一期仅钉钉 |
| 消息⊥通道 | 接收方只是通道配置 |
| 触发 | 类 DS 能力全要 |
| 调度 | 混合 C = 自建(B) + DS HTTP(A1) |
| 权限 | 一期全管理员；分级后置 |
| 成功标准 | A–F 闭环均必须 |
| 技术栈 | Python |

### 明确后置

- 权限 RBAC 细节
- 企微等其他通道实现
- 更多数据库插件
- 是否深度嵌入 DS 插件任务类型（当前以 HTTP 触发为准）
